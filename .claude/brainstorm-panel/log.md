## 2026-06-09 — Bootstrap app-wide adoption, re-eval (choosing between options)
- Proposed: UI/Frontend Architect (director), Design-system Guardian, Build/WebJars Engineer, Migration Strategist, Accessibility Advocate, Restraint Skeptic, QA (pre-seated). User added/removed: none.
- Style: swarm → director-led, 1 round (strong substantive convergence). Vote: **0 ADOPT-APPWIDE / 5 ADOPT-PARTIAL / 2 STAY-CUSTOM** (Architect + Skeptic).
- UNANIMOUS substance regardless of label: (1) **no big-bang 16-template rewrite**; (2) Bootstrap's real win is JS-wired components (modal/dropdown focus-trap) — UniTrack has **zero** today; (3) first modal → native `<dialog>` (0 deps) before Bootstrap JS; (4) IF adopting, **bridge-first**: `--bs-*: var(--app-*)` (app tokens master), toggle also sets `data-bs-theme`, ~20-30-line bridge lands before any component; keep theme tokens, Chart.js var-reads, status badges, `.gate/.bar/details.failure/.snippet/pill-subnav`.
- Only real split = **timing** (start incremental now vs wait for a trigger). Trigger to justify full adoption: ≥2 focus-managed JS components, OR a responsive/mobile need, OR ≥2 active contributors. None true today.
- Recurring (4th run): Restraint Skeptic wins dependency-scope — BUT this run is a genuine 5-2 split; the disagreement is "start now vs wait," not "how." If the user keeps re-invoking Bootstrap, that itself may be the trigger (maintainer preference) — honor it via the bridge-first incremental path, not a big-bang.
- Note (durable tech): `org.webjars:bootstrap:5.3.8` (Boot BOM does NOT manage it — pin version); Spring Boot 4 serves `/webjars/**` by default (add-mappings on); assets ship in-jar (offline-good for self-hosted); CSS-only ~30KB (skip `bootstrap.bundle.min.js`/Popper until a modal/dropdown lands); skip `webjars-locator` (hardcode versioned path, single-source via a `${bootstrap.version}` property); keep emoji (no Bootstrap Icons). Bridge needs `-rgb` token forms set literally per-theme. Chart.js does NOT re-render on theme toggle today (pre-existing, not a migration bug).

## 2026-06-09 — Client-side result-pushing / CI uploader (design → epic+tickets)
- Proposed: DevEx Director, Competitive Analyst (web research), Release/Packaging Engineer, SDET, Polyglot Ecosystem Advocate, Security/Supply-chain, YAGNI Skeptic. User added: **QA** ("add qa person"). User removed: none.
  - Signal (3rd consecutive design target the user adds a QA/consumer seat): **graduate — seat QA + SDET by default on design/product targets here.** QA(upload-reliability) and SDET(pipeline-author) were genuinely distinct and both pulled weight (QA: fail-loud-on-0-match + a real exit-code contract; SDET: PR head-SHA vs merge-SHA, `if: always()`, shard runKey).
- Style: swarm → director-led, converged in **1 round** (7/8 aligned). Director-led fits a single-vision DevEx target.
- Outcome: auto-detect CI metadata in the bash script (kill hand-wired `$GITHUB_*`) + reliability hardening + a composite Marketplace Action wrapping the SAME script + per-ecosystem recipes; Docker fast-follow; ingest-scoped token (server). Rejected with explicit triggers: Go binary, Maven/Gradle plugins, npm/pip pkgs, `curl|bash` installer, OIDC.
- **User gate-edits (overruled the panel's defer + reshaped the architecture):** the panel deferred a compiled binary and Maven/Gradle plugins and proposed bash as the core. The user steered: (1) a **Java JAR uploader** is the "most versatile in the long run … can lead to Gradle/Maven plugin," and (2) **"Docker / action can run [the] Java agent."** → Converged on **the JAR (`unitrack-cli`) as the single upload engine**, with Docker (JRE+JAR), a Docker-based GitHub Action, and the Maven/Gradle plugins all just *running/wrapping that one JAR*; bash drops to a minimal dependency-free fallback. Key lesson: the YAGNI objection to a binary was *"a second toolchain,"* which **does not apply to a JAR** (same JVM build the repo already has) — and a single JVM engine behind every delivery vehicle is less duplication than parallel bash+JAR. For THIS repo, default to a JAR-based core for any "client tool" target.
- Recurring (now 3×): **the YAGNI/Restraint skeptic wins the dependency-scope call** — minimal set over a sprawl; always capture "build later, only if X" exit ramps. (Caveat from this run: scope-skepticism must distinguish *new-toolchain* cost from *same-toolchain* cost.)
- Note (domain): competitor pattern = thin Action wrapping a binary + auto-detected metadata (Codecov/Datadog/Coveralls/Sonar); UniTrack's gap was the Action + auto-detect, not transport. CI-docs footguns to always flag: .NET TRX ≠ JUnit (needs `JunitXml.TestLogger`), JaCoCo XML is opt-in, `jest-junit` isn't default, Go needs `gotestsum` + `gocover-cobertura`. The shipped example workflow inlined raw `curl` (divergent from `scripts/unitrack-upload.sh`) — make the script the single source of truth.

- Style: swarm → director-led, converged in **1 round** (unusually strong consensus — 6/7 aligned on both decision and design). Director-led fit: a nav wants one coherent visual owner.
- **Framework decision**: user asked to weigh **Bootstrap via WebJars**; panel voted 6 effective STAY-CUSTOM / 0 genuine adopt (IA abstained as stack-agnostic; QA's "ADOPT" was actually "adopt a shared fragment"). Reason: ~30KB + a SECOND theme system (`data-bs-theme` vs the app's `data-theme`) + a SECOND token set (`--bs-*`) duplicating already-hand-rolled `.card/.badge/.table`, for one nav. WebJars mechanics are fine (ships in-jar — good for self-hosted); revisit only as a deliberate app-wide migration IF a forms/modals/grids wave arrives. Don't smuggle a framework in under a cosmetic ticket.
- Recurring: the **Restraint Skeptic keeps winning** the dependency-scope argument (consistent with YAGNI vetoing RBAC/MCP earlier). The cheap, token-reusing custom path is the house style.
- Note (repo UI conventions for future targets): the design system is ONE hand-rolled `app.css` with CSS-variable dark/light theming via `data-theme` on `<html>`. New components should consume existing tokens (`--panel/--panel-2/--border/--radius/--muted/--blue`), add an active state via a controller-passed key + `th:classappend` + `aria-current`, and live in `fragments/common.html` as a parameterized fragment reused across sibling pages. Verify visual changes with the self-skipping `@Tag("ui")` Selenium screenshot test **SubnavScreenshotUiTest** (kept on purpose for future UI work — screenshots land in `target/screenshots/`).
